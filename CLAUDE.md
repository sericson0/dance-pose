# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

TAngle (*Tango + Angle*) — an interactive 3D tool for teaching Argentine tango: pose two dancers (leader/follower) with anatomically constrained joints, view skeleton/body/muscle layers, and read off biomechanics (center of gravity, base of support, balance margin, joint angles, A/B pose comparison). Stack: Vite + Three.js, vanilla JS ES modules, no framework, no tests, no vite config file.

All three mesh layers are imported GLBs whose parts are re-parented onto our own joint rig (`src/skeletonMesh.js`); `main.js` loads them with top-level `await` before building the figures and falls back to procedural geometry on failure. The skeleton and muscle views are anatomical atlases (CC-BY-SA — distributing them requires attribution + a compatible license); the muscle GLB shares the skeleton's atlas frame, so it is loaded only when the skeleton did and bakes with the skeleton's scale. The body view is a pair of clothed Microsoft Rocketbox avatars (`man.glb` / `woman.glb`, MIT) — skinned meshes whose Biped bones are retargeted onto our joints at load. See `public/models/ATTRIBUTION.md` for all licenses.

Honours DEV_URL

Spread and rigDrift are the sharp instruments: a large but steady gap is bone-under-skin, a gap that grows with flexion is the rig/atlas divergence. **rigDrift is the only metric that can see the palm-hold class of bug** — how far the landmark wanders inside its own rig joint node across poses (the skeleton and clothed hands both ride the ATLAS wrist, so they drift together and their mutual gap reads a flawless 4 mm while the palm moves 136 mm inside `wrist_R`, which is what the embrace was aiming at). Bone-under-skin landmarks are `abs: null` — gap reported, NOT gated, because the true soft-tissue thickness is a real unknown non-zero distance and any threshold there is a number invented to be passed. Both scripts print `No console errors.` on success. They use `window.__app` (exposed at the bottom of `src/main.js`) — the scripted API for selection (`selectJoint`), posing (`editJoint`, `setJointDegrees`), chain mode (`setChainMode`), visibility (`setVisibleFigures`), camera, etc. Write new verification scripts in this same pattern. The Edge path is hardcoded to `C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe`.

## Architecture

Everything is **data-driven**: each body part, muscle, bone, joint limit, and movement is a row in a table, editable without touching the rest.

- `src/skeletonDef.js` — single source of anatomical truth: joint hierarchy (positions as height fractions, Drillis–Contini), per-axis joint limits in degrees, de Leva mass table (`MASS_SEGMENTS`) used for COG, IK chains, closed-chain anchor map (`ANCHOR_FOR`), dancer-vs-dancer collision capsules (`COLLIDERS`), and `BODY_PARTS`/`PART_OF_NODE` highlight groups. **Only the left side is defined; the right side is auto-mirrored.** The sole corner tables (`FOOT_CORNERS_*`/`TOE_CORNERS_*`, the balance footprint) are **sized to the rendered shoe of the man avatar** (measured from the skinned mesh — the rig's own toe segment overshoots the shoe, so the toes joint sits near the shoe TIP and the toe-pad corners only a little past it); a figure whose avatar wears a different shoe scales them per-figure via the `soleScale` constructor option (the follower's shorter heeled shoe uses `front: 0.78`, set where she is constructed in `main.js`).
- `src/skeletonMesh.js` — loads the imported anatomical skeleton GLB (`public/models/skeleton.glb`, CC-BY-SA — see `public/models/ATTRIBUTION.md`) and classifies each of its ~144 named bones onto one of our joint nodes (`classifyBone`). The GLB is Draco-compressed (decoder self-hosted in `public/draco/`) and holds only right-side + axial bones; the left side is mirrored at bake time. Matching is alphanumeric-normalized (GLTFLoader sanitizes names) and side is read from the bone's GLTF group, not the name suffix. Also loads the muscle GLB (`public/models/muscles.glb`, same source/license) via `loadMuscleMeshes`, routing ~60 named main-mover bellies to joint nodes with `classifyMuscle` (an exact normalized-name → `{ node, insert? }` map: `MUSCLE_NODE` gives the bone a belly rides, `MUSCLE_INSERT` names the far joint a bi-articular belly also crosses so it can skin/deform between the two — see `figure.js`); all shipped muscles are right-side and mirror to the left. The abdominal wall (rectus abdominis + obliques) rides the `pelvis` node (its caudal iliac/pubic attachment) and skins toward the `chest` with a full-length **spread** blend (`TRUNK_SHEETS` in skeletonMesh.js → the `spread` path in figure.js): the belly shears progressively from its pelvis end to its chest end, so a chest-vs-pelvis twist (tango dissociation, which lives almost entirely at the `chest` joint) stretches it on one side and shortens it on the other instead of carrying it along rigidly. It still highlights with the Torso part (`ride: 'spine'`). Also loads the clothed body avatars (`public/models/man.glb` / `woman.glb`, Microsoft Rocketbox, MIT) via `loadBodyMesh`, and exports `BODY_RETARGET` — the table mapping each 3ds Max Biped bone to the joint it snaps to, the bone/joint pair that defines its alignment direction, and its axial-stretch / foot-squash flags (see `Figure.#buildMeshBody`).
- `src/anatomy.js` — **fallback** procedural bone/muscle geometry from data tables, used only if a GLB fails to load: `LONG` plus small builders (skull, ribcage, pelvis, spine, hand, foot), and the muscle spindles (`MUSCLES_LEFT` / `MUSCLES_CENTER`).
- `scripts/build-muscles.mjs` — one-off asset pipeline (gltf-transform, run manually): trims the AnatomyTOOL "Upper limb" / "Lower limb" / "Muscles of thorax, abdomen and back" source GLBs down to the classified main-mover muscles, flattens materials, merges all three (deduping bellies that appear in more than one source, so the trunk model contributes only the abdominal wall), and Draco-compresses to `public/models/muscles.glb`. Imports `classifyMuscle` so its keep-list can't drift from the runtime.
- `scripts/build-body.mjs` — one-off asset pipeline for the body avatars: takes an FBX2glTF conversion of a Microsoft Rocketbox avatar (FBX → GLB comes out untextured because the FBX references 3ds Max map names), decodes the source TGA textures itself, downsizes them to 1024², wires them onto the body/head/opacity materials, and Draco-compresses to `public/models/man.glb` / `woman.glb`. The skin (bones + weights) passes through intact.
- `src/figure.js` — assembles a dancer as an Object3D joint tree (not a skinned mesh) with three toggleable mesh layers (skeleton / body / muscles); `addMesh(node, mesh, layer)` is the attach point for any geometry. `#buildMeshSkeleton()` scales the atlas bones to the figure's height, bakes each into its joint node's local frame (mirroring right→left, reversing winding), and merges per node+material into one mesh to keep draw calls low. **Limb bones (and their muscles) ride a parallel display-only "atlas" sub-tree, not the rig nodes.** The rig's anthropometric joint nodes (Drillis–Contini, for correct COG) sit a few cm off the imported skeleton's own joint centers, so a bone baked rigidly on a rig node pivots about the wrong point and swings visibly apart from its neighbour as a joint bends (a deeply-bent elbow opened a ~5 cm gap). `#buildAtlasNodes()` builds `this.atlasNodes` — an Object3D per seated limb joint, placed at the atlas joint center (`calibration.rest`) and slaved to the rig joint's local rotation by `syncAtlasNodes()` (called from every pose-mutating path: `setPose`/`resetPose`/`setJointDegrees`/`app.editJoint`, plus `main.js`'s loop) — so bones hung there pivot about the anatomical joint and stay welded in any pose. `#seatNode(name)` returns the atlas node if seated, else the rig node; the bone/muscle bakes and `#alignEndpointGeometry` all resolve through it. A whole limb chain must be seated together (a bone welds to its neighbour only if both ride matching trees); **all four limb chains are seated**: the arms (scapula → shoulder → elbow → wrist, the scapula staying a rig node as the chain anchor) and the legs (hip → knee → ankle → toes, the pelvis as the rig anchor). A heeled figure's atlas ankle/toes seats are raised by its `heelLift` (mirroring the rig ankle lift) so they sit exactly where the retargeted shoe's foot bones snap, and `#applyHeel` resolves its pitch target and the foot meshes' parent origins through `#seatNode`. The atlas fans the hand's fingers ~1.7× wider than the clothed avatar carries them, so `#handDesplayRotations()` adducts index/ring/pinky toward the middle finger at bake time (fraction `HAND_DESPLAY`, tuned to the avatar's own spread) so the bare skeletal fingers sit inside the glove instead of overshooting it. `#buildMeshMuscles()` does the same bake/mirror for the muscle GLB using the *skeleton's* atlas scale, keeping muscles as individual named meshes (each carries `userData.muscleName`). A belly that only rides one bone is parented rigidly to its joint; a bi-articular belly (one with an `insert`, e.g. biceps, gastrocnemius, rectus femoris) is baked in figure-local space, hung off `group`, and **skinned between its two joints** — `#addSkinnedMuscle` computes a per-vertex weight along the origin→insertion axis, and `updateMuscleSkin()` re-blends the baked positions/normals from the two joints' current transforms every frame (called from `main.js`'s loop and from `setPose`; it no-ops unless the muscle layer is visible). This is CPU linear-blend skinning, not `THREE.SkinnedMesh`, so the muscle's own frame never moves and the skin stays correct as the whole dancer translates/turns. `#buildMeshBody()` attaches the clothed avatar: it keeps the GLB's `THREE.SkinnedMesh` + skin weights but re-parents each Biped bone onto the matching joint node with a constant local matrix — onto the **atlas** node (`#seatNode`) for seated limb joints, so the clothed arm pivots about the same anatomical joint as the skeleton/muscle layers and stays welded to them through any bend (a rig-node pivot fanned the layers apart — a bent arm split the skeleton ~18 cm off the sleeve), and onto the rig node for everything unseated (torso/legs). To make the three layers coincide (so they overlay when toggled together), the body is retargeted onto the *atlas* rest pose, not the rig's own rest — the skeleton and muscle layers render the atlas pose (arms slightly abducted, hands/feet splayed), so `#atlasLimbRest()` estimates each limb joint's atlas position from the loaded skeleton bones (the meeting point of adjacent bone clusters). This estimate is computed once per build into `this.calibration` — the **single neutral rest** the body retarget and the endpoint roll both consume, so all layers reference one meeting point instead of re-deriving it inline. Each Biped bone snaps its origin to that atlas joint, aligns its bind direction along the atlas segment, and stretches axially to reach it (torso joints, which already agree, stay on the rig rest). A world-Y squash on foot/toe bones keeps shoe soles — heels included — grazing y = 0 at rest. Unmapped bones (fingers, face, clavicles) ride their re-parented ancestor unchanged, and the mesh hangs off `group` in 'attached' bind mode so all deformation is GPU-skinned with no per-frame CPU cost. `#alignEndpointGeometry()` then rolls the skeletal hands onto the clothed hands' orientation (the atlas holds the hand rolled differently from the avatar's). `#endpointAlignR` fits the *optimal* rotation (`hornRotation`, Horn's quaternion method) that overlays six corresponding regions at once — the five finger clusters and the palm block — using atlas-bone centroids for the skeleton and dominant-Biped-bone skin centroids for the body; the palm is weighted up since the two hands are shaped differently and no rigid rotation makes them fully coincide (frame-from-axes constructions fail here because the opposable thumb sits off the palm plane and tilts any single normal differently in each layer). It's applied as a rotation of the wrist-node mesh, and returns null (no roll) if the regions can't be gathered. `#measureHandMesh()` records each clothed hand's **mesh truth** — where the visible hand really is and how it is really turned — in `this.handMesh`, and it is the only honest source for either. Orientation (`fingers` along the middle finger's proximal phalanx, `palm` = the direction a curl first sweeps the fingertips, plus the `offset` quaternion from the rig-canonical hand frame) is measured in the **rig wrist's** frame, which is valid because the rig and atlas wrists share their local rotations and so agree in orientation exactly (measured: 0.0° apart in every pose); embrace.js conjugates its clasp/pronation targets by it, and `palmDirWorld(side)` / `fingerDirWorld(side)` expose the visible palm normal and finger aim. **Position is a different frame and a different story**: `center` (the palm's middle, midway between the avatar's own hand-bone origin and its knuckle row) is stored in the **atlas** wrist's frame via `#seatNode`, because that is the node the clothed hand is welded to, and `palmPosWorld(side)` reads it back. The two wrists share rotations but NOT positions, so they drift apart as the arm flexes — 6 cm with the arm hanging, **~18 cm at the embrace's -120° elbow**. Anything that POSITIONS a hand off `wrist_*`/`hand_*` is therefore aiming a phantom: it is what let the embrace solve a flawless palm-to-palm clasp on the rig nodes while the rendered hands hung ~15 cm apart in mid-air, invisible to a whole verification suite that also only ever measured rig nodes. Use `palmPosWorld`, never the rig wrist→hand midpoint. A heeled figure (`heelRise`, a fraction of stature — the follower's heels) raises its ankle node so the shoe's heel shows natively instead of being squashed flat, and `#applyHeel()` pitches the skeletal foot heel-up about the ball to sit inside the shoe and drops the balance corners (per-figure `footCorners`/`toeCorners`, which `lowestPointY` and `analysis.js` prefer over the flat `FOOT_CORNERS`) back to the shoe sole. Body highlight is a no-op on the avatar (one continuous skin, `userData.noHighlight`). Also owns `clampToFloor()` (runs every frame) and `setHighlight()`. The live calibration (`calibration.rest` + `calibration.endpointR`) is checked against a frozen snapshot every load by `#assertCalibration()` — a *tripwire* that `console.error`s (never throws) if a model or tuning constant has moved the layers apart. `calibrationJSON()` serializes it scale-free for baking.
- `src/rigCalibration.js` — **generated** (`npm run bake:rig`, via `scripts/build-calibration.mjs`): the frozen skeleton↔mannequin calibration, keyed by avatar (`man`/`woman` have different Biped binds). `rest` = canonical joint centers as fractions of stature; `endpointR` = skeletal-hand roll quaternions. Consumed only by `#assertCalibration()` as a drift tripwire (the live estimate stays the source of truth); the hard pass/fail gate is `scripts/dev-verify-calibration.mjs`. Re-bake whenever a GLB or a retarget constant changes.
- **`#atlasLimbRest` locates the shoulder and hip from the CHILD side**, not the parent. Its general rule — "the verts of this bone nearest the parent cluster's centroid" — is sound while the parent is a limb bone (the humerus's centroid sits up the arm, so the nearest radius/ulna verts really are the elbow), but it fails badly where the parent is a torso cluster: the chest's and pelvis's centroids lie far away and medial and drag the estimate down and inward. Measured against the atlas GLB's OWN bones (humerus 333 mm, radius 251, femur 474, tibia 377), the old estimate put the upper arm at 259 mm and the thigh at 418 mm — a 74 mm error at the shoulder and 56 mm at the hip, and it made the estimated forearm LONGER than the estimated upper arm, which is anatomically impossible. Since this rest positions the atlas display tree *and* drives the body retarget, every layer was pivoting the limb about a joint centre that far from the real head of the bone (all layers agreed with each other, because they all hang off the same wrong node — which is why no layer-vs-layer gate could see it). Shoulder and hip are now the far end of their own bone, away from the next joint down: upper arm 303 mm, thigh 430 mm, and the palm's rig-node drift fell 136 → 107 mm with the follower's `acromion` drift 39.5 → 26.2 mm (spread 21 → 6.8 mm).
- `src/landmarks.js` — **named anatomical points, located independently in each layer.** A landmark carries a separate recipe per layer (a bone-name regex plus a reducer: centroid, extreme along a figure-local axis, or nearest/farthest from a joint), so "the layers agree" becomes a real measurement. This is deliberately NOT what `dev-verify-alignment.mjs` measures: that compares the centroid of everything hanging on a joint node against the centroid of the skin attributed to it, and those are different objects (the skeleton's `spine` node owns the whole vertebral column and ribcage; the avatar's Spine bone owns a patch of skin), so its ~150 mm torso numbers are largely ownership artifact rather than misalignment. **Torso landmarks must be palpable** — acromion, iliac crest, C7 spinous, jugular notch — because elsewhere in the trunk bone sits deep to skin and a large gap is anatomy; limb landmarks are mostly endpoints for the same reason. A landmark's vertex SET is frozen at bind (rest) so it stays the same material point; its POSITION is read live, so it resolves in any pose. Two traps encoded here: `getVertexPosition` on a SkinnedMesh in 'attached' bind mode returns **figure-local, not world** (the model matrix is applied at render time — `dev-verify-alignment.mjs` reads it raw and is only correct because it zeroes `group.position` first), and an axis-based pick on a *sided* landmark must mirror x for the right side or the two sides silently select opposite edges of the same bone (measured: a 300 mm acromion gap that was pure recipe). Bone-level selection against the merged skeleton meshes works because `#buildMeshSkeleton` records `mesh.userData.boneRanges`. Also owns **`ENDPOINT_FITS`** — the corresponding REGIONS that `Figure.#endpointAlignR` runs its weighted similarity fit (Umeyama/Horn) over, so the fit is data like everything else and shares the landmark recipe language (a region is just a landmark that keeps many verts instead of a few). A region is only VALID if its two recipes describe the same sub-object, so the centroids are concentric; and the regions must not be collinear, or the roll about that line is unconstrained. Hand regions are `raw: true` (sampled from the un-baked atlas bones, `perBone` verts each) — **this deliberately preserves the frozen calibration**: reading the rendered geometry instead is arguably more correct, since the rendered hand carries the `HAND_DESPLAY` finger tuck, but switching rotated the frozen hand roll 9° and moved `palm_center` 4→8 mm, so it is left as a separately testable question rather than a refactor side effect. **The foot uses `mode: 'axis'`, not a similarity fit** — what matters for a foot is that it POINTS the way the shoe does, so the fit aligns the MIDLINE (dominant principal axis) of the skeletal foot with the shoe's and does nothing else. The fit is: stretch along the midline to match LENGTH (one axis only), rotate the midline onto the body's, and place the two midlines' **segment midpoints** together. Measured: the skeletal foot was **18.8° off** (leader) / **16.7°** (follower) → now 0.1° / 1.7°; foot `tipGap` 42.7 → 22–26 mm; skeletal foot reach now matches the shoe's (leader 173 vs 177, follower 134 vs 136, from 159/177 and 157/136). Three failed approaches are recorded in the table comment so they are not retried. (1) A hand-style *similarity* fit made it worse (`heel_back` 21→55 mm): a glove follows a hand so their regions are concentric, but a SHOE does not follow a foot (heel block, toe box), and a UNIFORM scale distorts width to fix length — the feet differ in length far more than width. (2) Aligning the two CENTROIDS slid the foot down its own length until its farthest-from-ankle point flipped from toe to heel — 290 mm of error, **while `measureAxes` reported a perfect 0.1°**, because a metric that measures what the fit forces is circular; the independent check that caught it is `dev-verify-alignment.mjs`'s foot `tipGap`. (3) Centroids still failed *after* length-matching (312 mm): a centroid is a mass average and a bone cloud bunches differently from the shoe skin around it, so the two never correspond. The **segment midpoint** is purely geometric, so with equal extents matching midpoints also matches both ENDS. On a heeled figure `heel_back` legitimately reads ~48 mm (calcaneus vs the shoe's heel BLOCK, which hangs below and behind it) against ~11 mm for the leader's flat shoe — it is informational, not gated; aim and `toe_tip` are the foot's real tests. The axis fit also subsumes the pitch `#applyHeel` used to bisect for (a heeled shoe's midline is already pitched), so `#applyHeel` now only drops the balance corners — note it assigns `mesh.quaternion` outright and would erase the fit if its pitch were restored.
- `src/ik.js` — analytic two-bone IK for open-chain limb dragging, and the closed-chain math (`editWithAnchor`, `pinAnchor`: pin a distal foot/hand, move the body above via rigid root compensation); feet-to-floor helper.
- `src/embrace.js` — the tango embrace as per-frame constraints (re-applied from `main.js`'s loop; UI "Embrace" section → `app.setEmbrace`), modelled on the real embrace. "Hold embrace (arms)", open side: the leader's LEFT hand holds the follower's RIGHT palm to palm around a clasp point stored in both chest frames (live target = their midpoint, so it follows the couple; engaging the checkbox seeds it at the canonical hold — out past the open-side shoulders, midway between the chests, at the "Clasp height" (`app.setClaspHeight` → `Embrace.setClaspHeight`; default shoulder level, a natural close-embrace frame with the elbows hanging, the slider raises it toward a salon frame) — the horizontal placement chosen so the clasp orientation is anatomically reachable, and dragging an open-side hand re-captures it. The elbow height follows the clasp height, not the finger tilt: the hand sits above the shoulder only when the clasp is raised, and that is what lifts the elbow. The palm surfaces touch (`PALM_GAP` between centers) **along the couple axis** (leader chest → follower chest, horizontal): his palm faces her, hers faces him — both medial facings a real wrist can deliver; a normal that asks a close-in palm to face laterally away from its own chest lands in the pronation dead zone at every swivel and the hand turns its back on the partner (the original clasp bug). All hand-orientation targets (clasp and closed-side `#pronate`) are expressed in the figure's **mesh-truth hand frame** (`Figure.handMesh`), not the rig's canonical wrist axes: the Biped bind carries each avatar palm rolled ~110° about the forearm from the rig's palms-forward wrist-local +Z, so aiming the node axis at the partner showed them the back of the hand (the leader clasp flip) while the node-frame numbers read perfect; `Figure.palmDirWorld(side)` is the visible palm normal dev-verify-embrace checks. Hand **positions** are mesh truth for the same reason: `handCenter(figure, side)` is the single choke point and it returns `Figure.palmPosWorld` (the visible palm center), never the rig wrist→hand midpoint — the clothed hand rides the atlas wrist, which sits ~18 cm from the rig wrist at the embrace's own deep elbow flexion, so solving on rig nodes joined two phantom hands perfectly while the rendered ones hung ~15 cm apart. `handCenter` calls `syncAtlasNodes()` before reading, and must: the IK inside `#solveArm` writes rig rotations directly without one, so an unsynced read returns the palm for the previous iteration's pose and turns the solve's fixed-point correction into a divergent feedback loop (measured: the hands flew 122 cm apart). `#solveArm` seeds and re-aims its wrist target with the **currently measured** rig-wrist→palm offset each pass — that offset is large and pose-dependent, not the old half-a-hand-along-the-fingers guess. Both joined hands aim for a vertical-handshake finger direction: fingers up, tilted from vertical by the "Clasp tilt" UI slider (`app.setClaspTilt` → `Embrace.setTilt`) toward the couple's midline, plus a soft `CLASP_WRAP_DEG` lean of each hand's fingers over the back of the partner's (at the default shoulder-height clasp the leader's open hand is limit-bound and leans up-and-in rather than straight up — the tilt has full authority only in the raised salon frame), and the clothed avatars' finger bones curl lightly (`Figure.setFingerCurl` at `CLASP_CURL`, tables in figure.js). The clasp hand orientation is solved exactly: fingers + palm pin the whole hand frame, decomposed to elbow pronation + wrist flex/deviation as YXZ Euler (both branches tried), the arm's swivel is chosen by sampling the elbow's circle (`#swivelForPalm`), and when the geometry is strained the fingers lean over in the palm plane (`#LEANS`) — candidates are ranked by the **achieved** palm/finger error after limit-clamping (palm weighted double), never by raw limit violation. Closed side: BOTH palms rest on the partner's back — his right on her back below her left shoulder blade (his arm passing UNDER her left arm), and her left high on his back behind his right shoulder blade, her arm draped OVER his and around his shoulder (`CLOSED_TARGETS`, fractions in the partner's chest frame). Hers is not a fixed point: it lerps out to `FOLLOWER_OPEN_REST` (his deltoid) by `openness()` as the chests separate from close-embrace to salon distance, because reaching AROUND him costs far more arm than resting on the near side of him and her arm is the shorter of the two — at salon distance the back point sits ~3 cm past her fingertips, and an out-of-reach embrace goal does not degrade into a shortfall, it ratchets the shoulder into its `-170°/-30°` limit corner and stands the arm straight up over her head (measured: palm 45 cm off, 38 cm too high). `MAX_REACH` in `#reachClamp` is the backstop for the same failure — the far-end twin of the long-standing `MIN_REACH` clamp, since both ends are the same rule (a goal the arm cannot occupy makes the per-frame solve ratchet instead of settle). Both closed-side hands `#pronate` toward the PARTNER'S CHEST NODE — the torso axis is square to the back surface wherever on it the hand landed; aiming hers at the midpoint of his upper arm (which is roughly where her hand already was) made the direction near-degenerate and left her palm 82° off the surface it was supposed to be lying on. All four arms re-solve every frame via `solveTwoBone`; closed-side arms keep their authored swivel (`embraceArms` pose in presets.js gives the elbows-down hang and the over/under layering) — don't force an elbow roll there, it fights joint limits on these close-in reaches and destabilizes the per-frame solve. "Close embrace": holds the horizontal chest-to-chest distance at `CONTACT_FRACTION`·(H₁+H₂) by sliding a dancer; distance-only, so the contact point can rotate around either torso (giros, calesita). Both constraints skip whatever the user is editing (`embraceEditing()` in main.js) and move the partner instead; editing an open-side arm re-captures the clasp so the partner tracks the user's hand. The closed side only solves while the couple roughly face each other (`#facing()`) — otherwise those rest points sit behind the shoulder's range and the per-frame solve would ratchet the arms into overhead poses (same reason `#solveArm` clamps its goal to `MIN_REACH` from the shoulder; `closedGoalWorld` exposes the clamped rest point, which in close embrace is where the follower's palm actually lands on his deltoid).
- `src/pins.js` — **contact pins**: user-authored per-frame constraints holding a spot on one dancer against a spot on the other (custom holds, a parada's foot contact). Each spot is stored in the local frame of the joint node nearest the click (`nearestJointNode`), so it rides that body part. The partner of whoever is being edited adapts (embrace convention; default the follower), and HOW depends on where the adapting end rides: an arm spot re-solves that arm via two-bone IK, a leg spot re-solves the leg (then re-clamps the mover to the floor), a torso/head spot slides the whole dancer horizontally. `maintainBody` (translation pins) runs with the embrace torso pull — before collision and the floor clamp, which push back; `maintainLimbs` runs after `maintainHands`, so a pin on an embrace arm deliberately overrides the embrace for that arm. Authoring: the topbar "Pin spots" mode (`app.pinClick` — first click sets a pending spot, a click on the other dancer completes the pin); `app.addPin` is the scriptable path; the sidebar "Contact pins" section lists/releases them. End markers turn amber with a strain line when anatomy can't close the gap; spots placed inside the partner's collision volume legitimately rest at surface contact (collision wins — do not "fix" this).
- `src/collision.js` — dancer-vs-dancer body collision as a per-frame constraint (runs in `main.js`'s loop between the torso-contact pull and the floor clamp): each dancer is the `COLLIDERS` capsule set from skeletonDef.js — the body `core` (torso/head/legs/feet, `group:'core'`) plus thin `arm` capsules (upper arm + forearm, no hand, `group:'arm'`). Any penetration deeper than ~2 mm is resolved by sliding the partner of whoever is being edited (default: the follower yields, same convention as the embrace) horizontally along the deepest contact's normal. The slide length is found by bracketing + bisecting the whole capsule-set clearance — interleaved limbs (crossed shins, a foot between the partner's feet) make clearance along a slide non-monotone, so fixed-step pushing can oscillate and never settle; the bisection lands the surfaces just touching (±0.5 mm), so contact is preserved, never repelled. **Arm capsules are colliders ONLY for the one arm the user is actively posing** (`resolveBodyCollision`'s `editedArm` arg, computed in main.js from the edited joint), and only against the partner's core — so dragging an arm into the partner can't pass through them, it displaces them, exactly like the body core does; the resting/wrapping embrace arms are deliberately NOT colliders (they lie ON and AROUND the partner by design — colliding them all the time shoves the couple apart and wrecks the authored presets), and arm↔arm never collides (the closed-side arms cross by design). The hand is never a collider (it is what the embrace rests on the partner). The trunk radii sit just inside the close-embrace `CONTACT_FRACTION` so held torso contact rests at exact surface touch without the two constraints fighting; head/shoulder radii are calibrated so the shipped contact presets (close embrace, walk) reach the held distance with ~0 clearance. `bodyClearance`/`bodyContacts` (also `app.bodyClearance()`) report the tightest **body-core** pairs (arms excluded — an arm resting on the body is contact, not penetration) for verification and radius tuning.
- `src/analysis.js` — COG from segment masses, foot contacts, convex hull, balance margin, key angles; weight distribution between feet (`weightDistribution`: L/R split, support foot, heel/mid/ball, on-axis) and tango metrics (`tangoStats`: dissociation, step length, turnout). Couple stats combine both dancers.
- `src/presets.js` — starting poses. **Joint sign conventions are documented at the top of this file** — read them before authoring poses. Two things about the embrace presets are load-bearing and easy to undo by accident. (1) **An embrace preset must be authored AT the embrace's own contact distance and with the real tango lateral OFFSET** (Close embrace: chests 21.3 cm apart, follower ~14 cm to the leader's right). What decides whether the two heads pass cheek to cheek or meet head-on is that offset, NOT the neck yaw — a head capsule runs `head→headTop`, i.e. nearly along the yaw axis, so yawing barely moves it. Measured at contact distance, the head capsules overlap **7.9 cm at a 5 cm offset** and 3.1 cm at 11 cm; only ~14 cm clears them. Author it square and torso contact is not merely unreached but geometrically impossible, and the failure is not a gap — the close-embrace pull hauls, collision refuses, and the leftover tangential push **walks the follower around the leader** until the heads clear, which they only do once she is ~22 cm round on his LEFT, the wrong side of the embrace, settling 4 cm short of contact. That happened on every engage. Authored at the offset the pose is already the solution and engaging the constraints moves nobody. The couple settles at lat ≈ −14 cm from any authored offset between −11 and −14, which is the check that ~14 is the geometry's own equilibrium and not a tuned number. (2) **`embraceArms`'s arm angles are BAKED from the embrace solve, not hand-authored** — apply the preset, tick both constraints, let it settle, dump the eight arm joints. Hand-authored angles went stale invisibly when the arm chain moved onto the anatomical joint centres, because the constraints overwrite the arms as soon as they are engaged; with them OFF, which is how the app STARTS, the stale angles were what the user saw — both dancers with arms flung out, clasp hands 32 cm apart, the leader's right palm 48 cm in front of his own chest instead of around her back. Re-bake after changing the solve or a placement.
- `src/draw.js` — floor annotations for teaching diagrams (topbar **Draw** mode): lines, arrows, circles, and text drawn flat on the dance floor as chalk-like meshes (no depth write + polygon offset, and they sit just above the balance hull/trail lines, so nothing z-fights). Each annotation carries its parameters in `userData.annotation`; the rubber-band preview of a half-authored shape lives in a separate `previewGroup` so committed drawings never mix with it. Look tuning (chalk color, stroke width, arrow head, text size) is the constants at the top.
- `src/main.js` — scene, picking, gizmos, interaction modes (rotate joint / drag hand-foot / move / turn / pivot-on-foot / pin spots / draw), undo stack, balance visuals, wooden floor; **keyboard nudges** drive whatever is active (one small change per keydown, so the OS key-repeat makes holding continuous; Shift = coarser): Move/Step figures as before, a selected joint's free axes (↑↓ = X, ←→ = Z else Y, PageUp/Down = Y twist, through `app.editJoint` so closed chain holds), or the live drag handle — IK target / swivel pole / caress ring / hips — moved camera-relative in the floor plane + PageUp/Down vertical, re-solved by `applyHandleChange` (shared with the gizmo's objectChange); **Move-hips auto-selects**: entering the mode seats the hips handle on the last-touched (else first visible) dancer without a click — clicking the other dancer moves it, empty-space clicks keep it, and it re-seats when its dancer is hidden or a preset is applied; A→B pose interpolation (`lerpPose`, `applyInterp`, `playInterp`) with a COG floor trail — every interpolation path (`applyStatesT`: A→B, sequence, trail, video export) **grounds planted feet**: a foot on the floor at both ends of a segment stays connected to it throughout (holding its spot, or gliding along the floor when the keyframes place it differently), the leg's hip/knee/ankle re-solving from the lerped pose (`stateFeet`/`groundInterpFeet`; `app.interpGroundFeet = false` restores the raw lerp) — translucent ghost figures of snapshots (`setGhost`), and `pivotFigure` (rotate about the ball of the support foot; combines with "Move as couple" for a calesita). **Move mode slides AND turns from ONE gizmo** — no Slide/Turn toggle. `selectFigure` attaches TWO `TransformControls` to the figure's group: `tcontrols` in translate mode (the floor-plane slide arrows, as before) and a second `turnControls` in rotate-Y mode (a yaw ring, `size 1.0` so it encircles the arrows). Grab the arrows to slide, the ring to turn — the same handle does both, which is what the split toggle used to gate. The ring is a separate instance because three.js `TransformControls` shows one mode at a time; it has its OWN `dragging-changed` (which pushes undo history and calls `beginFigureTurn`) and `objectChange` (`applyFigureTurn`), since the main `tcontrols` handlers never fire for it. Which vertical axis a turn spins about is `app.movePivot` (`setMovePivot`, the "Turn about" dropdown — the one control that survived the toggle removal), resolved by `turnPivot()` and used by the ring and the arrow keys alike: `'foot'` the ball of the support foot (the ground-referenced tango pivot — ocho, giro — the default), `'cog'` the axis through the centre of gravity (balance-referenced; the only axis a turn on both feet can use, and with "Move as couple" it is the pair's SHARED COG from `coupleReport`, the axis a giro actually orbits), `'root'` the figure's own origin. The gizmo spins the group about its own origin, so the `figTurn` capture (drag start, in `beginFigureTurn`) records the pivot plus both dancers' start transforms and `applyFigureTurn` re-derives every frame absolutely from them — never incrementally, so a long drag cannot drift. The **movement sequence** (`app.seqStates` + `seqAdd`/`seqUpdate`/`seqMove`/`seqDelete`/`applySeqT`/`playSeq`; sidebar "Sequence" section) generalizes A→B to any keyframe chain: `applyStatesT(states, t)` lerps segment-wise at equal time per segment, the COG trail traces whichever chain `trailStates()` picks (the sequence when it has ≥2 keyframes, else A/B), and the current sequence persists in localStorage with JSON export/import. **Animation export**: `app.recordPlayback(states, name)` replays a keyframe chain while capturing the canvas via `MediaRecorder` (`captureStream`), downloading a `.webm` — the ⏺ buttons in the Compare (A→B) and Sequence sections. The **dissociation floor visual** (`DissociationViz`; View → "Dissociation" checkbox) draws each dancer's floor-projected hip axis (their color) and shoulder axis (white) under the pelvis with a translucent wedge sweeping the signed twist between them — the picture of the `tangoStats` dissociation number, clearest in the Top view. The **walk** (`app.stepFigure` → `beginStep`/`poseStep`/`finalizeStep`) is a gait-phase step: it ends on the classic contact pose (front foot landed heel-first, trailing foot peeled onto the ball, pelvis yawed ~6° into the step with the chest counter-yawed — tango dissociation; the yaw SUM is preserved so authored trunk twists survive) and animates there through a floor-caressing collection (`app.stepAnims`, advanced in the render loop; `app.animateSteps=false` snaps). `plantFoot` reaches a leg's ankle to a flat-sole target and pitches the foot **about its ground contact** (ball for a roll, heel for a heel-strike — the ankle target swings along that arc; the roll is capped so the leg stays out near its reach instead of folding the knee), then nudges the target until the foot's lowest sole corner (`Figure.footLowY`) sits exactly on the floor. The body's along-travel position is **anchored to the support foot each step** (ending `STEP_ADVANCE` of the stride past it, preserving the dancer's own stance offset so a leaning couple isn't snapped apart) — accumulating a fixed advance instead drifts the body behind the feet until the front landing is out of reach. A linked partner steps the initiator's stride AND the mirrored foot (his left pairs her right; same-letter feet sit on opposite rails and jam leg-on-leg). Mid-stride the close-embrace couple legitimately rests a few cm wider than standing contact where the interleaved thighs brush (`COLLIDERS`). The **toe caress** (`caressToe`; clicking a toes joint in Drag-limb mode starts it, `app.caressFoot` scripts it) drags a leg by a floor ring while the big toe's pad stays exactly ON the floor: flat sole under the body rolling up to a pointed toe as the leg reaches (lápiz/tendu), hip/knee/ankle/toes absorbing everything. The pinned contact is the toe-PAD center (all other sole corners share its plane or sit behind it), so the sole never penetrates and the caress can never lift the body; when joint limits stop the toe reaching the floor at the cursor, a bisection along the ray from below the hip keeps it grounded at the farthest reachable point (reachability is monotone on that ray — NOT along the drag path, which crosses a tucked-under-the-body zone the solver can't ground). The **Move hips** mode (`app.moveHips`; the topbar "Planted" checkboxes → `app.hipsPlant`, auto-set from floor contact on pick) drags a pelvis handle on all three axes: the upper body rides rigidly (no posture change), feet marked planted keep their world position AND sole orientation via leg IK (the closed-chain hip-height slider now routes through the same code), unplanted legs ride along. **A rise never pulls a planted foot off the floor** (`groundPlantedLeg`): as the hips go up the planted leg first straightens (knee/hip extend), then the foot rolls up onto its toe (the ankle plantarflexes — a relevé) so the front sole still rests on the floor, and once even a full relevé can't hold the foot down the pelvis stops rising there — a Newton back-off inside `moveHips` clamps the height to where the most-lifted planted foot still reaches (past the limit it rises ~1:1 with the pelvis). The relevé measures from a base frozen at grab time (`captureHipsPlantBase` → `hipsState.plantBase`, captured in the hips-handle `dragging-changed`), so a rise-then-lower drag reverses cleanly (the foot flattens back down) instead of ratcheting; one-shot callers (the hip-height slider, keyboard nudges, scripts) fall back to a live snapshot per call. Verified by `dev-verify-hips-rise.mjs`. Move hips also offers **Slide or Twist** (its own sub-toolbar → `app.setHipsTool`); Twist swaps the handle for a yaw ring driving `app.pivotHips`, **the ocho-pivot dissociation**: the pelvis yaws and the whole lower body goes with it — hips, legs and feet turn as one unit, which is free since they hang off the pelvis node — while the spine twists to pay for it, the chest and lumbar joints counter-yawing by the same total so the shoulders, head and embrace keep facing exactly where they were (the yaw-sum-preserving idiom the walk's dissociation uses; exact upright, drifting slightly under a large pelvis tilt where the two yaws sit in non-parallel frames). The pelvis yaw is **clamped to the trunk's remaining counter-twist** (chest ±35° spent first — thoracic rotation is where tango dissociation lives — then the lumbar spine's ±8°, so ~43° from neutral, about a real dancer's range): past that the trunk has nothing left, and turning further would saturate the chest and start carrying the shoulders round with the hips, the one thing the move is defined by not doing. The "Planted" checkboxes belong to the SLIDE and are hidden in Twist — this move turns the feet on purpose. **Hover feedback** (rotate / drag-limb modes): the joint pick spheres are invisible at rest in body view AND sit inside the opaque avatar, so a glow alone never reaches the screen. Hovering a dancer there ghosts that dancer's whole joint set through the skin (`GHOST_OPACITY`) and lights the one under the cursor amber — `material.depthTest = false` is what draws them through the body, `renderOrder` keeps them above it, and `Figure.setLayers` restores both so a layer switch can't strand spheres floating in front. Skeleton view keeps its flat 0.22 look and normal depth sorting (the spheres already sit over visible bones). `app.photoDataURL`/`app.capturePhoto` snapshot the canvas to PNG with every gizmo/handle hidden. **Draw mode** authors floor annotations two clicks at a time on the y=0 plane (Text: one click + prompt, oriented to read right-way-up from the current camera; Esc, a mode change, or a click off the floor abandons a half-drawn shape); scriptable via `app.addDrawLine/addDrawArrow/addDrawCircle/addDrawText` + `removeLastDrawing`/`clearDrawings` (`app.drawings` lists them, `app.draw` is the Drawings instance). Annotations are scene content, deliberately OUTSIDE the pose undo stack — the Draw toolbar's ⌫ Last / Clear manage them. **COG click-to-front**: clicking a COG ball toggles drawing that whole indicator (ball, drop line, floor marker) through the dancers (`BalanceViz.setFront`: depth test off + late render order — the ball otherwise hides inside/behind the bodies, where a click couldn't even find it); an in-front ball wins the click outright (it is what the user sees), otherwise the nearest of ball vs. joint/figure hit wins, so ordinary picking is unaffected. `app.setCogHighlight(on, which?)` / `app.cogHighlight()` script it.
- `src/ui.js` — side panel: the **View** section's layer picker is a single `#layer-mode` dropdown (`LAYER_MODES`), three mutually exclusive views rather than three checkboxes: `body` (the opaque clothed avatar alone), `skeleton` (bones alone), `muscle` (muscles over the skeleton — the bellies need the bones behind them to read as anatomy). Body never combines with the other two because it is opaque and hides them. Scripts drive it by setting `#layer-mode`.value + dispatching `change`; `Figure.setLayers` still takes the three booleans, so `dev-verify-alignment.mjs` can still ask for the skeleton+body overlay directly. Also: sliders, stats, A/B compare with movement scrubber + ghost toggles + ⏺ Record, the Sequence section (keyframe list with Show/⟳/↑/↓/✕ rows, whole-movement scrubber, Play, ⏺ Record video, JSON export/import, localStorage persistence), the Contact pins section (pin list + Clear all; authoring happens in the 3D view), pose library (localStorage + JSON export/import); the joint picker is a two-column grid (center joints as a chip row, then Left/Right PAIRED per row so the side lives in the column header and the list is half as long); the **Foot map** section draws a top-down canvas of one foot as a classic ink foot print — an hourglass sole (deep medial arch waist, rounded heel) whose forefoot is **beveled and oblique** — the metatarsal break runs diagonally (the 1st MTP head sits forward of the 5th) and each ball corner chamfers into the toe crease via an anchor riding the straight MTP→crease chord, which keeps the corner flat instead of rounding off under the Catmull-Rom smoothing. The crease sits only just ahead of the metatarsal heads, because the toes own nearly the whole toe region (a hallux is ~14% of foot length against a ~16% toe region) and the corner cage's toe region is barely half the ball width — a crease pushed further forward eats the space the pads need. Five tilted toe ovals sit ahead of it, spaced anatomically: each at a fixed cross-station of the ball width (a real toe row spans the full forefoot breadth, hallux centered ~16% in from the medial edge … little toe ~92%), tips on the retreating toe arc (hallux at the tip line down to ~56% of the toe region for the little toe, from anthropometric toe lengths), pad centers hung back from their tips, and long axes radiating toward the heel (the big toe near-parallel to the foot axis, lateral toes tilting toward the midline). The row clears the sole by **construction**, not tuned constants: the sole's front edge is modelled across the foot by its three front anchors, and the whole row shifts forward by ONE shared amount, so a stubbier shoe (the follower's `soleScale`) or a foreshortened pointed foot can't sink pads into the sole and the arc keeps its shape. It is all hung on that figure's fitted corner cage so it stretches/turns/foreshortens with the real foot — with dashed heel|arch and arch|ball separators clipped to the print, heel/arch/ball/toes section labels along the print's own axis (the arch label offset laterally by half the waist dent so it doesn't hang over the medial edge; both labels and dashes hidden when the print draws too small to hold them), and the COG drop point over it, green inside/red outside, heel/mid-foot/ball zone in the note, refreshed on the stats tick (the balance hull itself still uses the raw corners, so the verdict stays honest while the print is stylized). **The print is drawn in the SOLE'S OWN PLANE** — the foot viewed perpendicular to its sole — not projected flat onto the floor. Flattening foreshortens a pitched foot into a blob (measured: a pointed foot's 27.6 cm corner cage collapsed to 4.4 cm), which is what made the print stop resembling the foot on screen and threw off where the COG read against it. The frame is a Newell best-fit normal over the corner cage (the corners are only near-coplanar once the toes bend), `fwd` = heel→toe flattened into that plane, `latAxis = fwd × normal`. **When the sole is flat the normal IS +Y and this reduces exactly to the old floor projection**, so a standing figure draws byte-identically — that reduction property is the regression guard, not a tolerance. The COG is the 3D COG dropped along the same normal (again identical to the old floor drop when flat). The **balance verdict deliberately stays a FLOOR question**: `margin`/hull still come from the corners' floor projection, the same base of support the 3D view outlines, so the green/red reading is unaffected by the drawing change. When only part of the sole is grounded (heel up, pointed foot) the contacting corners are filled as a dashed green **contact patch** — the same region the 3D support outline shows — and the note says so. Orientation: toes up (v = heel→toe), and the lateral axis is **anatomically fixed per foot** — the print is a footprint on the floor seen from above with the foot pointing away, so a RIGHT foot always draws with its big toe/arch on the canvas LEFT (medial) and a LEFT foot on the right, each reading as the correct foot regardless of camera or which way the dancer faces. This falls out of the single `toUV` projection because the two feet's medial ("in") corners lie on opposite world sides of the foot axis; no per-side sign is needed. **Do NOT tie u to the camera** to make the dot match the on-screen left/right: mirroring the frame to match the view turns a right foot's print into a left foot's (tried and reverted). The COG dot shares this projection, so it always lands on that foot's true medial/lateral side — read the dot against the drawn big toe/arch, not against the 3D screen. **Verified**: a sweep of all 12 presets × both dancers × both feet found the dot on the correct anatomical side in 48/48 cases, and `u` is provably screen-right for a camera looking straight down (`right = up × back = (fx,0,fz) × (0,1,0) = (-fz,0,fx)`), so the frame IS a true bird's-eye view. Beware when comparing against the 3D viewport: seen from the FRONT a dancer's medial side appears on the viewer's right, but the map (a print on the floor seen from above) puts it on the canvas left — the two legitimately mirror, and that is not a bug. Note also that the drawn print is stylized and INSET from the corner cage (the arch waist pulls the medial border in by 0.26·ballW), while the dot and the green/red verdict are measured against the raw cage — so the dot can sit visibly outside the inked arch while still being inside the balance hull. Also owns the topbar Draw sub-toolbar (line/arrow/circle/text pick + ⌫ Last / Clear, shown only in Draw mode).

| which axes a whole-figure turn can spin about | `turnPivot` in `main.js` + the `#move-pivot` options in `index.html` — re-verify with `dev-verify-turns.mjs` |
| how far the hips can twist under a still chest | the `y` limits of `chest` / `spine` in `skeletonDef.js` — `pivotHips` derives its clamp from them, so there is no separate constant |
| a foot's balance footprint (support hull / floor contact) | `FOOT_CORNERS_*`/`TOE_CORNERS_*` in `skeletonDef.js` (sized to the man avatar's rendered shoe) and the per-figure `soleScale` arg where a figure is constructed in `main.js` (the follower's shorter heeled shoe: `front: 0.78`). The corner sets fully own foot↔floor contact — `CORNER_NODES` in `figure.js` keeps the rig's overshooting phantom toe endpoint out of `lowestPointY`, so don't add foot joints back to the clearance sweep. Re-verify with `dev-verify-controls.mjs` + `dev-verify-features.mjs` + `dev-verify-gait.mjs` |

## Behavioral rules (deliberate design decisions — do not "fix")

- **Selecting a joint is NOT editing it — it must never move anyone.** Which dancer yields to the embrace and collision (the partner of whoever is being edited) is decided by `embraceEditing()` in main.js, and it keys off `app.lastEditedFigure` / `editStamp` (set by `app.markEdit`), NOT off `app.selected`. This is deliberate: `embraceEditing` used to return `app.selected` outright, so merely CLICKING a joint on one dancer flipped who yields and jolted the partner on a click that changed no pose — measured, selecting the leader's elbow spun the follower's shoulder up to 220° and slid her body 376 mm. Clicking a joint to look at or start posing it is the single most common action here, so it is inert. Every path that ACTUALLY changes a pose calls `markEdit` (the gizmo `objectChange`, `editJoint`, `applyHandleChange`, `slideFigure`/`turnFigure`/`pivotHips`/`moveHips`/`setPelvisHeight`/`stepFigure`, and `setHandCurl`); a new mutation path must too, or the constraints won't know whose partner to move. Between edits the *figure* is remembered (so a drag ending doesn't hand the couple back to the default mover and jolt them again) but the *joint* is dropped (so an arm isn't left exempt from the embrace after the user let go); `applyPreset` clears it, since a preset places both dancers and no one is mid-edit. A verification that simulates a move by writing `group.position` directly (bypassing every `markEdit` path) must call `app.markEdit` itself to stand in for the real move — see `dev-verify-collision.mjs`'s edited-push case.
- **Feet are free.** Feet may point, rise to demi-pointe, or lift off the floor. Never add logic that forces feet down; only floor *penetration* is blocked (`clampToFloor` lifts the whole dancer just enough, and settles back only by undoing its own lifts). "Feet to floor" is an explicit opt-in button (`app.groundFeet`); it also flattens the toes joints. Demi-pointe: point the ankles (x>0) and extend the toes (`toes_L/R` x<0) so the toe pads stay flat on the floor. The one deliberate exception: keyframe interpolation grounds a foot that is planted at BOTH ends of the segment (it holds its spot or glides along the floor — see `groundInterpFeet` in main.js); a foot airborne at either end still lerps freely, so kicks and boleos fly.
- **Open vs. closed kinetic chain** is a core teaching feature: open chain rotates everything below the joint (ordinary FK); closed chain keeps the distal end (planted foot / joined hand) fixed and moves the body above. **Selecting a joint always starts in open chain** — closed chain moves the BODY rather than the limb, which surprises you if you didn't ask for it, so it stays a deliberate opt-in on the legs/pelvis via the joint panel's toggle (there is no auto-detect from foot contact; that was removed). The anchor is picked automatically from `ANCHOR_FOR`; the toes joint anchors itself, so extending it in closed chain is a relevé (heel rises over the pinned ball of the foot). The pelvis hip-height slider is special-cased (`app.setPelvisHeight`): rigid root compensation cancels a pure translation exactly, so in closed chain the legs re-solve via IK instead — planted feet keep their place and sole orientation while the body sinks or rises.
- **A figure's facing is `figureYaw(figure)`, never `group.rotation.y`.** They agree only while nothing has written the group's quaternion — and the Move-mode rotate gizmo writes it every pointermove. Three's XYZ Euler decomposition of a pure yaw past 90° comes back as `[180°, 180° − yaw, 180°]`: the same orientation, but `rotation.y` then reads a *different angle* and, worse, **runs backwards when incremented** (Euler `[180, θ, 180]` = `R_y(180 − θ)`, so `rotation.y += d` subtracts d from the real facing). Measured: a gizmo turn to 103.6° reported 76.4°, and the missing 27° went straight into the turn's pivot compensation as 111 mm of support-foot drift — with the pivot arithmetic itself provably correct, which is why it read as a mystery. `figureYaw` reads the yaw off the quaternion; `canonicalizeYaw` puts the rotation back in `[0, yaw, 0]` form (lossless — the orientation is a pure yaw either way) and is called at turn-drag start and on every `objectChange`, so every OTHER consumer of `rotation.y` (arrow-key turns, `rotateAbout`/`pivotFigure`, preset placement, pose save/load) stays correct without each having to know. Anything new that writes a figure's quaternion must canonicalize too.
- **Joint limits are enforced everywhere** — gizmo, sliders, IK, and presets must all stay within the ranges in `skeletonDef.js`.
- **Two frames, two jobs — pick deliberately.** `Figure.worldPos(joint)` is the RIG joint centre: Drillis–Contini anthropometry, and the right input for **biomechanics** (`computeCOG` lerps de Leva segment masses between exactly these points; the joint-angle and balance readouts are defined on them). `Figure.surfacePos(joint)` / `surfaceNode(joint)` is the node the **visible body** is welded to — the atlas node on a seated limb joint, the same rig node elsewhere — and is the right input for **contact**: collision volumes, pin spots, anything asking where the dancer physically is. The difference is not subtle, measured across six poses: the visible palm sits 165 mm from the rig wrist and wanders 107 mm inside it as the arm flexes, but sits a steady 39 mm from the ATLAS wrist and wanders **0.0 mm**; the elbow is 97 mm / 111 mm against 37 mm / 15 mm. A capsule or pin built on the rig node is displaced from its own limb by more than that limb's radius. `dev-verify-frames.mjs` reports both (`rigDrift`/`nodeOff` vs `atlasDrift`/`atlasOff`), so this is checkable rather than assumed.
- **A spot stored in one frame must be replayed in that same frame.** `pins.js`'s `spotNode()` is the single rule deciding which frame a pin end lives in, and `nearestJointNode`, main.js's authoring, `endWorld` and `maintainLimbs` all route through it. Storing a click in one frame and playing it back in another puts the pin somewhere the user never clicked — measured at ~10 cm on a flexed arm. Also: `solveTwoBone` writes RIG rotations, so any loop that re-reads an atlas-framed spot between solver passes must `syncAtlasNodes()` first, or the residual correction reads the previous iteration's pose and diverges (the same trap documented for the embrace's `handCenter`).
- **THE ARM CHAIN IS COLLAPSED ONTO THE ANATOMICAL JOINT CENTRES, AND THE EMBRACE IS KNOWINGLY BROKEN PENDING REWORK.** The arm rig node and the anatomical node are now the same point, so the arm has ONE frame: `palm_center`'s drift inside its own rig node went 107 mm → **0**, `elbow_skin` 111 → 17 mm, and `handMesh`'s old split (orientation measured in the rig wrist, position stored in the atlas wrist) is gone. Everything else holds — calibration matches, preset grounding unchanged (`low` ≈ 0), collision, pins (arm 0.6 cm), muscles including the bi-articular biceps/triceps skinning, and the foot fits. **The embrace does not**, and that was an accepted, deliberate trade (it needed rework anyway). Its exact failing state, as the baseline to fix against: `close embrace` clasp gap **3.8 cm** (want 1.4) and closed-side palms off by **L 17.6 cm** (the follower's side of this is FIXED — she now rests on his back at 0.2–0.4 cm settled, 5.3 cm in this transient, via the sliding `CLOSED_TARGETS` rest point and `#reachClamp`; the leader's 17.6 is the part still outstanding, and the suite is at least DETERMINISTIC now — it used to return 1.7, 16.9 and 50.9 cm on three identical runs of this same case); after a 60° pivot, clasp gap **11.6 cm** with the leader's clasp fingers **61°** off the tilt direction. **Fix `#solveArm`, not `embraceArms`** — re-authoring those angles for the new rest was measured to move the closed-side error only 17.8 → 17.6 cm, because the closed-side arms are IK-solved and the authored angles merely seed the swivel; the real gap is the solve against the shoulder's new position (15 mm medial, 15 mm lower). The other consequence of the new rest is already handled: it is **abducted ~9.1° with a 19.4° elbow bend** where the old rest hung straight down and collinear, which broke `solveTwoBone`'s assumption that `rotation.x = 0` is full extension — `REST_BEND` in `ik.js` derives each hinge's rest bend from `JOINTS` and subtracts it (a no-op on the still-collinear legs).
- **Arms are migrated to surface nodes; legs deliberately are not.** Both `collision.js` (arm capsules) and `pins.js` (arm spots) resolve through `surfaceNode`; legs stay on rig nodes even though they are seated too. Legs are coupled to things the arms are not — `clampToFloor` and the balance hull read the sole corners in the RIG ankle's frame, and the `COLLIDERS` radii are calibrated so the shipped contact presets rest at ~0 clearance. Driving a leg in the atlas frame fights the floor clamp (measured: it lifted a pinned foot from 9 mm off the floor to 66 mm). Migrating legs means moving the corner tables with them and re-verifying grounding — a separate change, not an oversight.
- **Measure the hand the VIEWER sees, never the rig node.** The clothed hand is welded to the *atlas* wrist (`Figure.palmPosWorld` / `palmDirWorld` / `fingerDirWorld`); the rig `wrist_*`/`hand_*` nodes agree with it in rotation but drift up to ~18 cm from it in position as the arm flexes. Constraints solve the rig chain — that is correct, it is the IK effector — but every *goal* and every *measurement* must be mesh truth, with the residual between them corrected each pass. The cautionary tale: an embrace solved entirely on rig nodes reported a perfect 2.2 cm palm-to-palm clasp, backed by a green verification suite, while the rendered hands hung 15 cm apart in mid-air — a suite built on rig nodes could not see it in principle, and no amount of tuning inside the solve (pronation range, swivel search, lean ranking) could either, because the objective pointed at a hand that was not being drawn. Verification of anything touching a hand must read the avatar's own bones (`fingerBones`), not the rig. The same divergence is worth suspecting anywhere else limb *positions* are read off rig nodes on a flexed limb — arm collision capsules and arm contact pins are unaudited on this point.
- **The dancers may touch but never occupy each other's space** (`collision.js`). Walking, kicking or dragging one dancer into the other displaces the partner — that is the point (a leader entering the follower's space moves her, like a real sacada) — and the edited dancer always keeps their ground. Never "fix" a pose by letting bodies overlap: if a contact pose won't reach, either the pose is dishonest (e.g. heads nodding through each other — fix the pose, see the Walk preset's neck angles) or a `COLLIDERS` radius is too fat. Close embrace pulls to `CONTACT_FRACTION` *or* until bodies rest surface-on-surface, whichever comes first — after a pivot the couple may legitimately rest foot-against-foot farther than chest contact.
- **A contact pin holds spots together, never through each other.** Pins adapt the partner of whoever is being edited (arm/leg pins re-solve that limb within its joint limits, torso pins slide the dancer horizontally); collision still wins, so a pin whose spots sit inside the partner's volume legitimately rests at surface contact with a small residual spot gap — don't "fix" that by disabling collision, and an out-of-reach pin shows its amber strain line instead of teleporting anyone. A pin on an embrace arm overrides the embrace hold for that arm by design (`maintainLimbs` runs after `maintainHands`).
- **A posed arm can't pass through the partner's body, but the embrace arms rest freely on it.** The arm being actively posed is a collider against the partner's core, so pushing it into them displaces them (never "fix" this by letting the arm spear through); but arms are colliders *only while being posed* — the embrace and presets rest arms ON and AROUND the partner by design, so making them collide otherwise shoves the couple apart (which is why this is edit-scoped, not always-on — do not "simplify" it to always-on). The hand is never a collider (the embrace rests it on the partner).
- Pose-authoring gotcha: pelvis x-tilt or deep toe-points can push a foot below the floor, which makes the clamp lift the dancer and ruins the balance stats (the support foot floats and the weight reading lands on the wrong foot). Author trailing/pointed feet with hip/knee/ankle angle triples whose toe grazes y ≈ 0 — don't derive them by hand; probe candidate angles with `scripts/dev-tune.mjs` until `low` ≈ 0 and the standing sole is flat at 0. Known-good triples at standing pelvis heights: trailing grazing toe `hip 15 / knee 36 / ankle 33` (pelvisY 0.527), collected free foot beside the support foot `hip 2 / knee 50 / ankle 42` (pelvisY 0.525).
