# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

TAngle (*Tango + Angle*) — an interactive 3D tool for teaching Argentine tango: pose two dancers (leader/follower) with anatomically constrained joints, view skeleton/body/muscle layers, and read off biomechanics (center of gravity, base of support, balance margin, joint angles, A/B pose comparison). Stack: Vite + Three.js, vanilla JS ES modules, no framework, no tests, no vite config file.

All three mesh layers are imported GLBs whose parts are re-parented onto our own joint rig (`src/skeletonMesh.js`); `main.js` loads them with top-level `await` before building the figures and falls back to procedural geometry on failure. The skeleton and muscle views are anatomical atlases (CC-BY-SA — distributing them requires attribution + a compatible license); the muscle GLB shares the skeleton's atlas frame, so it is loaded only when the skeleton did and bakes with the skeleton's scale. The body view is a pair of clothed Microsoft Rocketbox avatars (`man.glb` / `woman.glb`, MIT) — skinned meshes whose Biped bones are retargeted onto our joints at load. See `public/models/ATTRIBUTION.md` for all licenses.

```

node scripts/dev-interact.mjs <outDir>         # selection, IK drag, A/B compare, screenshots + console errors
node scripts/dev-verify-features.mjs <outDir>  # presets stay grounded, weight/tango stats, interpolation, ghosts, pivot
node scripts/dev-verify-toes.mjs <outDir>      # toes (MTP) joint, demi-pointe, closed-chain relevé + pelvis height
node scripts/dev-verify-muscles.mjs <outDir>   # bi-articular muscles skin between their two joints (deform on bend), screenshots
node scripts/dev-verify-body.mjs <outDir>      # clothed avatars load + retarget, soles graze the floor, knees move feet not heads
node scripts/dev-verify-alignment.mjs <outDir> # skeleton-vs-body overlay: per-joint centroid drift + hand/foot endpoint gaps, translucent-body overlay screenshots
node scripts/dev-verify-calibration.mjs        # hard gate: live skeleton<->mannequin calibration matches the frozen rigCalibration.js snapshot (exits 1 on drift; re-bake with `npm run bake:rig`)
node scripts/dev-verify-embrace.mjs <outDir>   # embrace constraints: clasp joined (palm to palm along the couple axis, his palm facing her and hers him, fingers aimed up the tilted clasp direction, tilt slider re-aims — tested with the clasp raised, where fingers-up has authority; the default frame sits at shoulder height with the elbows hanging) through moves/turns/pivots, torso contact held (or resting body-on-body where collision stops the pull earlier)
node scripts/dev-verify-collision.mjs <outDir> # body collision: presets settle penetration-free, walking/kicking into the partner displaces them, edited dancer keeps ground, close embrace still reaches exact contact
node scripts/dev-verify-draw.mjs <outDir>      # Draw mode floor annotations (line/arrow/circle/text — scripted API + real two-click pointer authoring) and the COG click-to-front highlight; honours DEV_URL for a dev server on another port
node scripts/dev-tune.mjs '<json cases>'       # pose-authoring probe: prints foot-sole heights for candidate joint angles
```

Both scripts print `No console errors.` on success. They use `window.__app` (exposed at the bottom of `src/main.js`) — the scripted API for selection (`selectJoint`), posing (`editJoint`, `setJointDegrees`), chain mode (`setChainMode`), visibility (`setVisibleFigures`), camera, etc. Write new verification scripts in this same pattern. The Edge path is hardcoded to `C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe`.

## Architecture

Everything is **data-driven**: each body part, muscle, bone, joint limit, and movement is a row in a table, editable without touching the rest.

- `src/skeletonDef.js` — single source of anatomical truth: joint hierarchy (positions as height fractions, Drillis–Contini), per-axis joint limits in degrees, de Leva mass table (`MASS_SEGMENTS`) used for COG, IK chains, closed-chain anchor map (`ANCHOR_FOR`), dancer-vs-dancer collision capsules (`COLLIDERS`), and `BODY_PARTS`/`PART_OF_NODE` highlight groups. **Only the left side is defined; the right side is auto-mirrored.**
- `src/skeletonMesh.js` — loads the imported anatomical skeleton GLB (`public/models/skeleton.glb`, CC-BY-SA — see `public/models/ATTRIBUTION.md`) and classifies each of its ~144 named bones onto one of our joint nodes (`classifyBone`). The GLB is Draco-compressed (decoder self-hosted in `public/draco/`) and holds only right-side + axial bones; the left side is mirrored at bake time. Matching is alphanumeric-normalized (GLTFLoader sanitizes names) and side is read from the bone's GLTF group, not the name suffix. Also loads the muscle GLB (`public/models/muscles.glb`, same source/license) via `loadMuscleMeshes`, routing ~60 named main-mover bellies to joint nodes with `classifyMuscle` (an exact normalized-name → `{ node, insert? }` map: `MUSCLE_NODE` gives the bone a belly rides, `MUSCLE_INSERT` names the far joint a bi-articular belly also crosses so it can skin/deform between the two — see `figure.js`); all shipped muscles are right-side and mirror to the left. The abdominal wall (rectus abdominis + obliques) rides the `spine` node and skins toward the `chest`. Also loads the clothed body avatars (`public/models/man.glb` / `woman.glb`, Microsoft Rocketbox, MIT) via `loadBodyMesh`, and exports `BODY_RETARGET` — the table mapping each 3ds Max Biped bone to the joint it snaps to, the bone/joint pair that defines its alignment direction, and its axial-stretch / foot-squash flags (see `Figure.#buildMeshBody`).
- `src/anatomy.js` — **fallback** procedural bone/muscle geometry from data tables, used only if a GLB fails to load: `LONG` plus small builders (skull, ribcage, pelvis, spine, hand, foot), and the muscle spindles (`MUSCLES_LEFT` / `MUSCLES_CENTER`).
- `scripts/build-muscles.mjs` — one-off asset pipeline (gltf-transform, run manually): trims the AnatomyTOOL "Upper limb" / "Lower limb" / "Muscles of thorax, abdomen and back" source GLBs down to the classified main-mover muscles, flattens materials, merges all three (deduping bellies that appear in more than one source, so the trunk model contributes only the abdominal wall), and Draco-compresses to `public/models/muscles.glb`. Imports `classifyMuscle` so its keep-list can't drift from the runtime.
- `scripts/build-body.mjs` — one-off asset pipeline for the body avatars: takes an FBX2glTF conversion of a Microsoft Rocketbox avatar (FBX → GLB comes out untextured because the FBX references 3ds Max map names), decodes the source TGA textures itself, downsizes them to 1024², wires them onto the body/head/opacity materials, and Draco-compresses to `public/models/man.glb` / `woman.glb`. The skin (bones + weights) passes through intact.
- `src/figure.js` — assembles a dancer as an Object3D joint tree (not a skinned mesh) with three toggleable mesh layers (skeleton / body / muscles); `addMesh(node, mesh, layer)` is the attach point for any geometry. `#buildMeshSkeleton()` scales the atlas bones to the figure's height, bakes each into its joint node's local frame (mirroring right→left, reversing winding), and merges per node+material into one mesh to keep draw calls low. **Limb bones (and their muscles) ride a parallel display-only "atlas" sub-tree, not the rig nodes.** The rig's anthropometric joint nodes (Drillis–Contini, for correct COG) sit a few cm off the imported skeleton's own joint centers, so a bone baked rigidly on a rig node pivots about the wrong point and swings visibly apart from its neighbour as a joint bends (a deeply-bent elbow opened a ~5 cm gap). `#buildAtlasNodes()` builds `this.atlasNodes` — an Object3D per seated limb joint, placed at the atlas joint center (`calibration.rest`) and slaved to the rig joint's local rotation by `syncAtlasNodes()` (called from every pose-mutating path: `setPose`/`resetPose`/`setJointDegrees`/`app.editJoint`, plus `main.js`'s loop) — so bones hung there pivot about the anatomical joint and stay welded in any pose. `#seatNode(name)` returns the atlas node if seated, else the rig node; the bone/muscle bakes and `#alignEndpointGeometry` all resolve through it. A whole limb chain must be seated together (a bone welds to its neighbour only if both ride matching trees); **currently only the arms are seated** (scapula → shoulder → elbow → wrist, the scapula staying a rig node as the chain anchor) — the leg chain still bakes on the rig nodes (deferred: it entangles with the heeled-follower `#applyHeel` machinery). The atlas fans the hand's fingers ~1.7× wider than the clothed avatar carries them, so `#handDesplayRotations()` adducts index/ring/pinky toward the middle finger at bake time (fraction `HAND_DESPLAY`, tuned to the avatar's own spread) so the bare skeletal fingers sit inside the glove instead of overshooting it. `#buildMeshMuscles()` does the same bake/mirror for the muscle GLB using the *skeleton's* atlas scale, keeping muscles as individual named meshes (each carries `userData.muscleName`). A belly that only rides one bone is parented rigidly to its joint; a bi-articular belly (one with an `insert`, e.g. biceps, gastrocnemius, rectus femoris) is baked in figure-local space, hung off `group`, and **skinned between its two joints** — `#addSkinnedMuscle` computes a per-vertex weight along the origin→insertion axis, and `updateMuscleSkin()` re-blends the baked positions/normals from the two joints' current transforms every frame (called from `main.js`'s loop and from `setPose`; it no-ops unless the muscle layer is visible). This is CPU linear-blend skinning, not `THREE.SkinnedMesh`, so the muscle's own frame never moves and the skin stays correct as the whole dancer translates/turns. `#buildMeshBody()` attaches the clothed avatar: it keeps the GLB's `THREE.SkinnedMesh` + skin weights but re-parents each Biped bone onto the matching joint node with a constant local matrix — onto the **atlas** node (`#seatNode`) for seated limb joints, so the clothed arm pivots about the same anatomical joint as the skeleton/muscle layers and stays welded to them through any bend (a rig-node pivot fanned the layers apart — a bent arm split the skeleton ~18 cm off the sleeve), and onto the rig node for everything unseated (torso/legs). To make the three layers coincide (so they overlay when toggled together), the body is retargeted onto the *atlas* rest pose, not the rig's own rest — the skeleton and muscle layers render the atlas pose (arms slightly abducted, hands/feet splayed), so `#atlasLimbRest()` estimates each limb joint's atlas position from the loaded skeleton bones (the meeting point of adjacent bone clusters). This estimate is computed once per build into `this.calibration` — the **single neutral rest** the body retarget and the endpoint roll both consume, so all layers reference one meeting point instead of re-deriving it inline. Each Biped bone snaps its origin to that atlas joint, aligns its bind direction along the atlas segment, and stretches axially to reach it (torso joints, which already agree, stay on the rig rest). A world-Y squash on foot/toe bones keeps shoe soles — heels included — grazing y = 0 at rest. Unmapped bones (fingers, face, clavicles) ride their re-parented ancestor unchanged, and the mesh hangs off `group` in 'attached' bind mode so all deformation is GPU-skinned with no per-frame CPU cost. `#alignEndpointGeometry()` then rolls the skeletal hands onto the clothed hands' orientation (the atlas holds the hand rolled differently from the avatar's). `#endpointAlignR` fits the *optimal* rotation (`hornRotation`, Horn's quaternion method) that overlays six corresponding regions at once — the five finger clusters and the palm block — using atlas-bone centroids for the skeleton and dominant-Biped-bone skin centroids for the body; the palm is weighted up since the two hands are shaped differently and no rigid rotation makes them fully coincide (frame-from-axes constructions fail here because the opposable thumb sits off the palm plane and tilts any single normal differently in each layer). It's applied as a rotation of the wrist-node mesh, and returns null (no roll) if the regions can't be gathered. A heeled figure (`heelRise`, a fraction of stature — the follower's heels) raises its ankle node so the shoe's heel shows natively instead of being squashed flat, and `#applyHeel()` pitches the skeletal foot heel-up about the ball to sit inside the shoe and drops the balance corners (per-figure `footCorners`/`toeCorners`, which `lowestPointY` and `analysis.js` prefer over the flat `FOOT_CORNERS`) back to the shoe sole. Body highlight is a no-op on the avatar (one continuous skin, `userData.noHighlight`). Also owns `clampToFloor()` (runs every frame) and `setHighlight()`. The live calibration (`calibration.rest` + `calibration.endpointR`) is checked against a frozen snapshot every load by `#assertCalibration()` — a *tripwire* that `console.error`s (never throws) if a model or tuning constant has moved the layers apart. `calibrationJSON()` serializes it scale-free for baking.
- `src/rigCalibration.js` — **generated** (`npm run bake:rig`, via `scripts/build-calibration.mjs`): the frozen skeleton↔mannequin calibration, keyed by avatar (`man`/`woman` have different Biped binds). `rest` = canonical joint centers as fractions of stature; `endpointR` = skeletal-hand roll quaternions. Consumed only by `#assertCalibration()` as a drift tripwire (the live estimate stays the source of truth); the hard pass/fail gate is `scripts/dev-verify-calibration.mjs`. Re-bake whenever a GLB or a retarget constant changes.
- `src/ik.js` — analytic two-bone IK for open-chain limb dragging, and the closed-chain math (`editWithAnchor`, `pinAnchor`: pin a distal foot/hand, move the body above via rigid root compensation); feet-to-floor helper.
- `src/embrace.js` — the tango embrace as per-frame constraints (re-applied from `main.js`'s loop; UI "Embrace" section → `app.setEmbrace`), modelled on the real embrace. "Hold embrace (arms)", open side: the leader's LEFT hand holds the follower's RIGHT palm to palm around a clasp point stored in both chest frames (live target = their midpoint, so it follows the couple; engaging the checkbox seeds it at the canonical hold — out past the open-side shoulders, midway between the chests, at the "Clasp height" (`app.setClaspHeight` → `Embrace.setClaspHeight`; default shoulder level, a natural close-embrace frame with the elbows hanging, the slider raises it toward a salon frame) — the horizontal placement chosen so the clasp orientation is anatomically reachable, and dragging an open-side hand re-captures it. The elbow height follows the clasp height, not the finger tilt: the hand sits above the shoulder only when the clasp is raised, and that is what lifts the elbow. The palm surfaces touch (`PALM_GAP` between centers) **along the couple axis** (leader chest → follower chest, horizontal): his palm faces her, hers faces him — both medial facings a real wrist can deliver; a normal that asks a close-in palm to face laterally away from its own chest lands in the pronation dead zone at every swivel and the hand turns its back on the partner (the original clasp bug). Both joined hands aim for a vertical-handshake finger direction: fingers up, tilted from vertical by the "Clasp tilt" UI slider (`app.setClaspTilt` → `Embrace.setTilt`) toward the couple's midline, plus a soft `CLASP_WRAP_DEG` lean of each hand's fingers over the back of the partner's (at the default shoulder-height clasp the leader's open hand is limit-bound and leans up-and-in rather than straight up — the tilt has full authority only in the raised salon frame), and the clothed avatars' finger bones curl lightly (`Figure.setFingerCurl` at `CLASP_CURL`, tables in figure.js). The clasp hand orientation is solved exactly: fingers + palm pin the whole hand frame, decomposed to elbow pronation + wrist flex/deviation as YXZ Euler (both branches tried), the arm's swivel is chosen by sampling the elbow's circle (`#swivelForPalm`), and when the geometry is strained the fingers lean over in the palm plane (`#LEANS`) — candidates are ranked by the **achieved** palm/finger error after limit-clamping (palm weighted double), never by raw limit violation. Closed side: his right palm on her back below her left shoulder blade — his arm passing UNDER her left arm — and her left palm on his right deltoid, her arm draped OVER his (`CLOSED_TARGETS`, fractions in the partner's chest frame). All four arms re-solve every frame via `solveTwoBone`; closed-side arms keep their authored swivel (`embraceArms` pose in presets.js gives the elbows-down hang and the over/under layering) — don't force an elbow roll there, it fights joint limits on these close-in reaches and destabilizes the per-frame solve. "Close embrace": holds the horizontal chest-to-chest distance at `CONTACT_FRACTION`·(H₁+H₂) by sliding a dancer; distance-only, so the contact point can rotate around either torso (giros, calesita). Both constraints skip whatever the user is editing (`embraceEditing()` in main.js) and move the partner instead; editing an open-side arm re-captures the clasp so the partner tracks the user's hand. The closed side only solves while the couple roughly face each other (`#facing()`) — otherwise those rest points sit behind the shoulder's range and the per-frame solve would ratchet the arms into overhead poses (same reason `#solveArm` clamps its goal to `MIN_REACH` from the shoulder; `closedGoalWorld` exposes the clamped rest point, which in close embrace is where the follower's palm actually lands on his deltoid).
- `src/collision.js` — dancer-vs-dancer body collision as a per-frame constraint (runs in `main.js`'s loop between the torso-contact pull and the floor clamp): each dancer is the `COLLIDERS` capsule set from skeletonDef.js — the body `core` (torso/head/legs/feet, `group:'core'`) plus thin `arm` capsules (upper arm + forearm, no hand, `group:'arm'`). Any penetration deeper than ~2 mm is resolved by sliding the partner of whoever is being edited (default: the follower yields, same convention as the embrace) horizontally along the deepest contact's normal. The slide length is found by bracketing + bisecting the whole capsule-set clearance — interleaved limbs (crossed shins, a foot between the partner's feet) make clearance along a slide non-monotone, so fixed-step pushing can oscillate and never settle; the bisection lands the surfaces just touching (±0.5 mm), so contact is preserved, never repelled. **Arm capsules are colliders ONLY for the one arm the user is actively posing** (`resolveBodyCollision`'s `editedArm` arg, computed in main.js from the edited joint), and only against the partner's core — so dragging an arm into the partner can't pass through them, it displaces them, exactly like the body core does; the resting/wrapping embrace arms are deliberately NOT colliders (they lie ON and AROUND the partner by design — colliding them all the time shoves the couple apart and wrecks the authored presets), and arm↔arm never collides (the closed-side arms cross by design). The hand is never a collider (it is what the embrace rests on the partner). The trunk radii sit just inside the close-embrace `CONTACT_FRACTION` so held torso contact rests at exact surface touch without the two constraints fighting; head/shoulder radii are calibrated so the shipped contact presets (close embrace, walk) reach the held distance with ~0 clearance. `bodyClearance`/`bodyContacts` (also `app.bodyClearance()`) report the tightest **body-core** pairs (arms excluded — an arm resting on the body is contact, not penetration) for verification and radius tuning.
- `src/analysis.js` — COG from segment masses, foot contacts, convex hull, balance margin, key angles; weight distribution between feet (`weightDistribution`: L/R split, support foot, heel/mid/ball, on-axis) and tango metrics (`tangoStats`: dissociation, step length, turnout). Couple stats combine both dancers.
- `src/presets.js` — starting poses. **Joint sign conventions are documented at the top of this file** — read them before authoring poses.
- `src/draw.js` — floor annotations for teaching diagrams (topbar **Draw** mode): lines, arrows, circles, and text drawn flat on the dance floor as chalk-like meshes (no depth write + polygon offset, and they sit just above the balance hull/trail lines, so nothing z-fights). Each annotation carries its parameters in `userData.annotation`; the rubber-band preview of a half-authored shape lives in a separate `previewGroup` so committed drawings never mix with it. Look tuning (chalk color, stroke width, arrow head, text size) is the constants at the top.
- `src/main.js` — scene, picking, gizmos, interaction modes (rotate joint / drag hand-foot / move / turn / pivot-on-foot / draw), undo stack, balance visuals, wooden floor; A→B pose interpolation (`lerpPose`, `applyInterp`, `playInterp`) with a COG floor trail, translucent ghost figures of snapshots (`setGhost`), and `pivotFigure` (rotate about the ball of the support foot; combines with "Move as couple" for a calesita). **Draw mode** authors floor annotations two clicks at a time on the y=0 plane (Text: one click + prompt, oriented to read right-way-up from the current camera; Esc, a mode change, or a click off the floor abandons a half-drawn shape); scriptable via `app.addDrawLine/addDrawArrow/addDrawCircle/addDrawText` + `removeLastDrawing`/`clearDrawings` (`app.drawings` lists them, `app.draw` is the Drawings instance). Annotations are scene content, deliberately OUTSIDE the pose undo stack — the Draw toolbar's ⌫ Last / Clear manage them. **COG click-to-front**: clicking a COG ball toggles drawing that whole indicator (ball, drop line, floor marker) through the dancers (`BalanceViz.setFront`: depth test off + late render order — the ball otherwise hides inside/behind the bodies, where a click couldn't even find it); an in-front ball wins the click outright (it is what the user sees), otherwise the nearest of ball vs. joint/figure hit wins, so ordinary picking is unaffected. `app.setCogHighlight(on, which?)` / `app.cogHighlight()` script it.
- `src/ui.js` — side panel: sliders, stats, A/B compare with movement scrubber + ghost toggles, pose library (localStorage + JSON export/import); also the topbar Draw sub-toolbar (line/arrow/circle/text pick + ⌫ Last / Clear, shown only in Draw mode).

## Behavioral rules (deliberate design decisions — do not "fix")

- **Feet are free.** Feet may point, rise to demi-pointe, or lift off the floor. Never add logic that forces feet down; only floor *penetration* is blocked (`clampToFloor` lifts the whole dancer just enough, and settles back only by undoing its own lifts). "Feet to floor" is an explicit opt-in button (`app.groundFeet`); it also flattens the toes joints. Demi-pointe: point the ankles (x>0) and extend the toes (`toes_L/R` x<0) so the toe pads stay flat on the floor.
- **Open vs. closed kinetic chain** is a core teaching feature: open chain rotates everything below the joint (ordinary FK); closed chain keeps the distal end (planted foot / joined hand) fixed and moves the body above. The anchor is picked automatically from `ANCHOR_FOR`; the toes joint anchors itself, so extending it in closed chain is a relevé (heel rises over the pinned ball of the foot). The pelvis hip-height slider is special-cased (`app.setPelvisHeight`): rigid root compensation cancels a pure translation exactly, so in closed chain the legs re-solve via IK instead — planted feet keep their place and sole orientation while the body sinks or rises.
- **Joint limits are enforced everywhere** — gizmo, sliders, IK, and presets must all stay within the ranges in `skeletonDef.js`.
- **The dancers may touch but never occupy each other's space** (`collision.js`). Walking, kicking or dragging one dancer into the other displaces the partner — that is the point (a leader entering the follower's space moves her, like a real sacada) — and the edited dancer always keeps their ground. Never "fix" a pose by letting bodies overlap: if a contact pose won't reach, either the pose is dishonest (e.g. heads nodding through each other — fix the pose, see the Walk preset's neck angles) or a `COLLIDERS` radius is too fat. Close embrace pulls to `CONTACT_FRACTION` *or* until bodies rest surface-on-surface, whichever comes first — after a pivot the couple may legitimately rest foot-against-foot farther than chest contact.
- **A posed arm can't pass through the partner's body, but the embrace arms rest freely on it.** The arm being actively posed is a collider against the partner's core, so pushing it into them displaces them (never "fix" this by letting the arm spear through); but arms are colliders *only while being posed* — the embrace and presets rest arms ON and AROUND the partner by design, so making them collide otherwise shoves the couple apart (which is why this is edit-scoped, not always-on — do not "simplify" it to always-on). The hand is never a collider (the embrace rests it on the partner).
- Pose-authoring gotcha: pelvis x-tilt or deep toe-points can push a foot below the floor, which makes the clamp lift the dancer and ruins the balance stats (the support foot floats and the weight reading lands on the wrong foot). Author trailing/pointed feet with hip/knee/ankle angle triples whose toe grazes y ≈ 0 — don't derive them by hand; probe candidate angles with `scripts/dev-tune.mjs` until `low` ≈ 0 and the standing sole is flat at 0. Known-good triples at standing pelvis heights: trailing grazing toe `hip 15 / knee 36 / ankle 33` (pelvisY 0.527), collected free foot beside the support foot `hip 2 / knee 50 / ankle 42` (pelvisY 0.525).
